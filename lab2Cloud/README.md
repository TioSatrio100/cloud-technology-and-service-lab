# Лабораторная работа 2. Сравнение сервисов Amazon Web Services и Microsoft Azure. Создание единой кросс-провайдерной сервисной модели.

---

## 1. Цель работы
Целью данной работы является получение навыков аналитики спектра публичных облачных сервисов без привязки к конкретному вендору. Студент должен построить единую сервисную модель для двух облачных провайдеров (AWS и Azure) и освоить универсальную классификацию сервисов на основе их функциональности.

## 2. Исходные данные
В работе использовались:
* **Данные Лаб. работы №1:** Результаты классификации ресурсов AWS.
* **Биллинг Azure:** Файл `Azure lab team 9.csv`, содержащий подробные данные о потреблении ресурсов Microsoft Azure.
* **SQL-параметры:** Использование символа `%` в качестве подстановочного знака (wildcard) для сопоставления описаний сервисов с правильными категориями.

## 3. Выполненные задачи

### 3.1. Кросс-провайдерный анализ данных
Идентификация сервисов Microsoft Azure, функционально эквивалентных сервисам AWS из Лабораторной работы №1:
* **Compute (Вычисления):** Azure Container Instances соответствует AWS ECS/EC2.
* **Network (Сеть):** Azure Bandwidth и Application Gateway соответствуют AWS Route53/CloudFront.
* **AI & ML (ИИ и машинное обучение):** Cortana Intelligence соответствует Amazon Lex/Kendra.
* **Storage (Хранение):** Data Lake и SQL DW соответствуют Amazon S3/RDS.

### 3.2. Разработка логики классификации
Применение единой 5-уровневой иерархии для обоих провайдеров:
1. **IT Tower**: Основная инфраструктурная категория.
2. **Service Family**: Группировка по техническим функциям.
3. **Service Type**: Конкретный тип сервиса.
4. **Service Sub Type**: Детализация специализации.
5. **Service Usage Type**: Минимальная метрика потребления.

### 3.3. Автоматизация данных (Python/Pandas)
Процесс заполнения таблицы был выполнен с помощью Python-скрипта для обеспечения согласованности терминологии и точности сопоставления на основе ключевых слов в столбцах `Meter Category` и `Meter Name`.

## 4. Результаты
Сформирован итоговый файл, объединяющий метаданные AWS и Azure. Результаты подтверждают, что модель классификации, разработанная в Лаб. работе №1, универсальна и применима к другим облачным провайдерам (Azure).

## 5. Классификация сервисов (Единая модель)

### 5.1. Модель IaaS (Infrastructure as a Service)
* **Azure Container Instances & Registry**: Инфраструктура для запуска контейнеров.
* **Bandwidth & Application Gateway**: Базовые компоненты сети и балансировки нагрузки.

### 5.2. Элементы PaaS (Platform as a Service)
* **Azure SQL Data Warehouse**: Полностью управляемая аналитическая платформа баз данных.
* **Data Lake Store**: Сервис хранения данных для обработки Big Data.
* **Machine Learning Services**: Среда разработки моделей ИИ.

### 5.3. Элементы SaaS (Software as a Service)
* **Cortana Intelligence / Customer Insights**: Готовые аналитические программные сервисы.
* **Network Watcher**: Облачный инструмент для диагностики сети.

### 5.4. Специфические модели и стратегии ценообразования
* **Azure Reservations**: Аналог *AWS Savings Plans*, используемый для оптимизации затрат через обязательства по потреблению.

## 6. Вывод
В ходе выполнения работы было доказано, что модель облачных сервисов может быть **вендоро-независимой**. Использование единой структуры **IT Tower** и **Service Family** позволяет ИТ-менеджменту анализировать потребление ресурсов от разных провайдеров (AWS и Azure) в рамках единой отчетности. Это критически важно для избежания привязки к вендору (*vendor lock-in*) и упрощения аудита облачной инфраструктуры.
